Group Address Structure

What is a Group Address?

Group addresses (GAs) are the logical addresses used for communication between KNX devices. Unlike individual addresses (which identify a specific device), group addresses represent a function - like "Living Room Light Switch" or "Kitchen Blind Position".

Multiple devices can listen to the same GA, enabling one switch to control multiple lights, or multiple displays to show the same temperature.

3-Level Addressing (Main/Middle/Sub)

KNX uses hierarchical addressing: Main/Middle/Sub (e.g., 1/2/15)

LevelRangeBitsTypical Use
Main Group0-315 bitsMajor function category (Lighting, HVAC, Blinds)
Middle Group0-73 bitsLocation/floor or sub-function
Sub Group0-2558 bitsIndividual object/function

Total: 16 bits = 65,536 possible group addresses

2-Level Addressing (Main/Sub)

Alternative format: Main/Sub (e.g., 1/2055)

LevelRangeBits
Main Group0-315 bits
Sub Group0-204711 bits

Less common but useful for very flat structures.

Reserved & Special Addresses

AddressConventionRecommendation
0/x/xCentral/System functionsAvoid for normal devices; use for "All Off", central commands
14/x/xBuilding servicesSometimes used for HVAC central, building management
15/x/xDiagnostics/CommissioningReserved for ETS diagnostics in some projects
x/x/0"All in group" broadcastSome use sub-address 0 for group-wide commands
0/0/0Broadcast to allSpecial system address - never use for normal objects

Standard Structure

Before you start: Decide on your GA structure and document it. Changing later is painful.

The industry standard is function-based organization, which works for any installation size:

  • Main Group: Function type (Lighting=1, Blinds=2, HVAC=3, etc.)
  • Middle Group: Location/floor (Ground=0, First=1, etc.)
  • Sub Group: Specific object (sequential within each main/middle)

Standard 3-Level GA Structure

Overview

The professional standard for KNX group addressing follows a function-first hierarchy:

  • Main Group (1-31): What it does (Lighting, Blinds, HVAC...)
  • Middle Group (0-7): Where it is (Ground Floor, First Floor...)
  • Sub Group (0-255): Specific object/function

Note: Main Group 0 is reserved for central/system functions.

Standard Main Groups

MAIN GROUPS (by function):

Communication Flags (CRWTUI)

Every group object has flags that control how it communicates on the bus. Understanding these is essential for proper KNX operation.

Flag Definitions

FlagNameWhat it does
CCommunicationRequired for any bus activity. Without C, the object is completely isolated from the bus.
RReadObject responds to GroupRead requests. Essential for status objects that other devices query.
WWriteObject accepts GroupWrite commands. Required to receive commands from other devices.
TTransmitObject sends a GroupWrite when its value changes. Essential for sensors and status feedback.
UUpdateObject updates its internal value when it receives a GroupWrite. Used for displays and synchronization.
IRead on InitObject sends a GroupRead on device startup to get current value from bus. Useful for synchronization.

Common Flag Combinations

FlagsPatternUse CaseExample
C-W--Command InputReceives commands from other devicesSwitch actuator input, blind position input
CR-T-Status OutputSends current state, responds to read requestsSwitch status, sensor values, actuator feedback
C--T-Transmit OnlySends values but doesn't respond to readsPush button (sends on press, nothing to read)
CRWT-Full BidirectionalBoth control and status in one objectThermostat setpoint (can be set locally or remotely)
CRU--Display/MonitorReads values and updates from bus writesVisualization panel, touch screen display
CRWTUIFull FunctionalityAll operations enabledRarely used; typically too permissive

Common Mistakes

  • Missing C flag: Object won't communicate at all - nothing works
  • Missing R on status: Visualization can't read current state on startup
  • Missing T on sensor: Value changes but never sends to bus
  • Missing W on actuator input: Actuator ignores all commands
  • U flag on actuator command: Can cause feedback loops!

Flag Selection Guide

ACTUATOR (receives commands):
  Command input:  C-W--  (accepts writes)
  Status output:  CR-T-  (sends state, answers reads)

SENSOR (sends values):
  Value output:   CR-T-  (sends on change, answers reads)

PUSH BUTTON (triggers actions):
  Button press:   C--T-  (just sends, nothing to read)

DISPLAY/VISUALIZATION:
  Value display:  CRU--  (reads + updates from writes)
  Or with init:   CRUI-  (also reads on startup)

BIDIRECTIONAL (control point):
  Setpoint:       CRWT-  (full read/write/transmit)

Individual Address (Physical Address)

Every KNX device has a unique individual address (IA) that identifies it on the bus. Unlike group addresses (which are logical), the IA is the device's "name".

Format: Area.Line.Device

ComponentRangeBitsDescription
Area0-154 bitsBackbone segment / building area
Line0-154 bitsLine within the area
Device0-2558 bitsDevice on the line

Example: 1.2.15 = Area 1, Line 2, Device 15

Addressing Best Practices

  • Start at 1, not 0: Device 0 is reserved for couplers
  • Leave gaps: Use 1, 5, 10, 15... for easy expansion
  • Group by location: All devices in Room 1 on same range (1-9), Room 2 (10-19), etc.
  • Group by type: Or group all lights (1-20), all blinds (21-40), etc.
  • Document everything: Create an address list before installation
  • Label devices: Write the IA on each device during installation

Line Organization Strategies

STRATEGY 1: By Floor
  Line 1.1 = Ground Floor
  Line 1.2 = First Floor
  Line 1.3 = Second Floor

STRATEGY 2: By Function
  Line 1.1 = Lighting (all floors)
  Line 1.2 = Blinds (all floors)
  Line 1.3 = HVAC (all floors)
  Line 1.4 = Sensors (all floors)

STRATEGY 3: By Room Type (Commercial)
  Line 1.1 = Open Plan Office
  Line 1.2 = Meeting Rooms
  Line 1.3 = Common Areas
  Line 1.4 = Server Room / Technical

KNX Best Practices

üìã Planning Phase

  • Plan BEFORE buying: Design your GA structure, device locations, and line topology on paper first
  • Create a GA spreadsheet: Document every group address with its function, DPT, and linked devices
  • Choose a consistent naming convention: "Room-Function-Object" or "Function-Room-Object"
  • Draw the topology: Sketch areas, lines, and device placements
  • Calculate power requirements: Each line needs adequate power supply for connected devices
  • Leave 30% expansion room: You WILL want to add devices later

üîó Group Address Design

  • Separate command and status: Never use the same GA for both - it causes feedback issues
  • Use status GAs for ALL actuators: Visualization won't work without feedback
  • Leave gaps between sub-addresses: Room 1 uses 0-9, Room 2 uses 10-19, etc.
  • Reserve sub-address 0: Some use x/x/0 for "all in group" commands
  • Be consistent across the project: If dimmer brightness is always +2 from switch, keep it that way everywhere
  • Don't over-link: Only connect objects that NEED to communicate

üîß Configuration Tips

  • Set actuators to "Send on change": Status should update automatically when state changes
  • Enable "Read on Init" for visualization: Panels should request current state on startup
  • Configure cyclic transmission for sensors: Temperature should send every 5-10 minutes even if unchanged
  • Set appropriate send thresholds: Temperature sensor should send on 0.5¬∞C change, not every 0.01¬∞C
  • Test with bus monitor: Verify telegrams are sent correctly before finalizing

‚ö° Bus Performance

  • Max ~50 telegrams/second sustained: KNX TP1 runs at 9600 baud
  • Avoid telegram storms: Don't configure 50 devices to all transmit on the same trigger
  • Use delays for "All Off" scenes: Stagger actuator responses by a few ms
  • Separate heavy traffic: Put frequently-communicating sensors on their own line
  • Group lighting on same line: Reduces coupler traffic for local switching
  • Filter at couplers: Configure line couplers to only pass necessary GAs

üè† Residential Specifics

  • Every light needs a physical switch: KNX should enhance, not replace basic function
  • Plan for WAF (Wife Acceptance Factor): Keep UI simple, hide complexity
  • Central Off by front door: Essential - one button to turn off everything when leaving
  • Panic/All On scene: For emergencies, turn on all lights
  • Night mode: Limit maximum brightness after midnight
  • Presence simulation: Random light switching when away

üè¢ Commercial Specifics

  • Zone control: Group open-plan offices into controllable zones
  • Energy management: Monitor consumption, implement scheduling
  • Integration points: Plan BACnet/Modbus gateways for BMS integration
  • Emergency lighting: Ensure KNX doesn't interfere with statutory systems
  • Access control: Consider integration with building access
  • Maintenance access: Plan how technicians will access the system

üîç Troubleshooting Checklist

  • Device not responding: Check IA is programmed, check bus connection, check power
  • Commands not working: Verify GA is linked on both sender and receiver, check flags (W on receiver?)
  • Status not updating: Check T flag on sender, check R flag if using Read requests
  • Random behavior: Look for duplicate GAs, feedback loops, or faulty device
  • Slow response: Check bus load, look for telegram storms, verify couplers aren't overloaded
  • Device going offline: Check power supply capacity, measure bus voltage (21-30V DC)

üìù Documentation Requirements

  • GA list: Every group address with description, DPT, linked devices
  • Device list: Every device with IA, location, type, firmware version
  • Topology diagram: Areas, lines, couplers, device counts
  • Floor plans: Device locations marked on drawings
  • Scene definitions: What each scene does to which devices
  • Parameter settings: Non-default device parameters that affect behavior
  • Backup of ETS project: With all programming complete, stored safely
